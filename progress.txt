## Iteration 4 — US-004: Database models with UUID primary keys
- What was done: Created all SQLAlchemy 2.0 async models with UUID PKs, timestamp mixins, async session factory. 8 models: User, Category, Product, ListItem, Receipt, Purchase, PriceHistory, UserProductPreference. ListItem has all required fields (status, refresh fields, source, confidence, display_order). All relationships and FKs defined with proper cascade rules. Composite indexes for key query patterns (user+status, refresh engine, price lookups).
- Files affected:
  - backend/app/db/__init__.py (new)
  - backend/app/db/session.py (new) — async engine, session factory, get_db dependency with commit/rollback
  - backend/app/models/__init__.py (new) — re-exports all models
  - backend/app/models/base.py (new) — Base, UUIDMixin (gen_random_uuid), TimestampMixin (func.now)
  - backend/app/models/user.py (new) — User with Google OAuth fields
  - backend/app/models/category.py (new) — user-scoped categories with unique constraint
  - backend/app/models/product.py (new) — global products with barcode + normalized_name indexes
  - backend/app/models/list_item.py (new) — core entity with status cycling, refresh fields, enums
  - backend/app/models/receipt.py (new) — Receipt + Purchase models with JSON parsed data
  - backend/app/models/price_history.py (new) — price tracking with composite index
  - backend/app/models/user_product_preference.py (new) — per-user product preferences
- Learnings for next cycles:
  - Patterns: SQLAlchemy 2.0 Mapped[] + mapped_column() works well with mypy strict; use `from __future__ import annotations` + TYPE_CHECKING for forward refs to avoid circular imports; String columns with Python enums preferred over PG native enums for migration flexibility
  - Traps: mypy strict requires `dict[str, Any]` not `dict` for JSON columns; `type: ignore[assignment]` not needed when proper type annotation used
  - Helpful context: Tables created: users, categories, products, list_items, receipts, purchases, price_history, user_product_preferences; categories are user-scoped (each user gets own copy); products are global (shared across users)

- What was done: Initialized Alembic with async template, configured env.py to read DATABASE_URL from app.config.Settings, updated alembic.ini to skip hardcoded URL. Created manual migration 0001_initial_schema.py with all 8 tables (users, categories, products, list_items, receipts, purchases, price_history, user_product_preferences) including indexes, constraints, and proper FK cascade rules. Created seed script at app/db/seed.py with 15 default Hebrew categories and seed_categories_for_user() function for programmatic use.
- Files affected:

  - backend/alembic/env.py (new) — async env with app models imported for autogenerate

  - backend/app/db/seed.py (new) — 15 Hebrew categories seed with idempotent insert
- Learnings for next cycles:

  - Helpful context: conda env is "smart-shopping"; alembic is already in requirements.txt; migration uses gen_random_uuid() server_default matching the UUIDMixin; seed script can be run as `python -m app.db.seed <user_id>` or imported programmatically
